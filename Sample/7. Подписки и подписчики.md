# Создание подписчиков. Следование за пользователями

Модель данных нашего приложения имеет еще одну дополнительную
деталь, о которой нельзя не упомянуть: в отличие от симметричной дружбы в
стиле Facebook (или ВКонтакте), которая всегда взаимна (по крайней мере, на
уровне модели данных), взаимоотношения в стиле Twitter потенциально
асимметричны – Кальвин может читать сообщения Хоббса, при этом Хоббс
может не читать сообщений Кальвина. Чтобы определить различие между
двумя этими случаями, примем терминологию активных и пассивных
взаимоотношений: если Кальвин читает сообщения Хоббса, но не наоборот, у
Кальвина с Хоббсом активные взаимоотношения, а у Хоббса с Кальвином –
пассивные. 

Здесь также добавлен составной индекс, отвечающий за уникальность
пары (follower_id/followed_id), чтобы один пользователь мог последовать за
другим только один раз.
1. Теперь убираем (или комментируем) включение файла footer.php в
каждом файле проекта. При это надо убедиться, что все js скрипты
подключены в конце каждого документа.
2. Создаем в базе данных новую таблицу relationships. Данная таблица
будет иметь три атрибута: id (стандартная реализация), follower_id,
followed_id. Здесь также добавляется составной индекс UNIQUE, отвечающий
за уникальность пары (follower_id/followed_id), чтобы один пользователь мог
последовать за другим только один раз. На уровне программирования при
попытке подписаться будет выдавать ошибку. 

3. Задаем внешний ключ к таблице users. 

4. Теперь, когда таблица имеет столбцы можно заполнить таблицу
тестовыми данными (связями)

5. Смысл здесь такой, что id 1 следует за 3 и 2, а 3 имеет подписчика 1 и
2 имеет подписчика 1. Каждая пара значений должна быть уникальна. Все id
это номера в таблице users, которые связаны с таблицей relationships. Чтобы
лучше понять надо задать себе правильные вопросы. Во-первых, что
создается, когда пользователь начинает читать сообщения другого
пользователя? Во-вторых, что уничтожается, когда пользователь перестает это
делать? Поразмыслив, мы видим, что в этих случаях приложение должно
создать либо удалить взаимоотношение между двумя пользователями. То есть
пользователь связан множеством взаимоотношений с читаемыми (или
читателями) через эти взаимоотношения.
6. Так как количество Followers и Followed будет отображаться в файле
profile.php, то работать будем с ним. Для каждого зарегистрированного
пользователя будет свой profile.php. Он будет иметь вид
http://sample/profile.php?id=1
Здесь значение 1 получено из GET запроса (запрос, который отдает
данные). Эти данные хранятся в суперглобальном массиве $_GET. Таким
образом, когда мы зашли на страницу пользователя, мы должны запомнить его
id. Можно для этого завести специальную переменную $id. 

В таблице relationship находим все записи, где значение столбца
follower_id = id пользователя и followed_id = id пользователя. В листинге 1
обозначим $current_id = $user ->id

Следует заметить, что команда R::find на выходе дает массив бинов,
значения которых можно получить в цикле foreach. Теперь можно посчитать
количество читаемых пользователей и читателей. Ниже в листинге 2 в
переменных-счетчиках $i и $j будут накапливаться значения.

Следовательно, создайте предварительно в проекте стандартные файлы
followed.php и followers.php с подключением базы данных и хидера.
Содержимое followed.php такое же, как и profile.php за тем
исключением, что вместо количества сообщений и самих сообщений
выводится специальная таблица, которая показывает id тех пользователей,
которых читает данный пользователь с возможностью перейти на профиль
этих пользователей 

```html
<table class="table table-bordered">
 <thead>
 <tr>
 <th min-with="10px">id</th>
 <th min-with="10px">name</th>
 </tr>
 </thead>
 <tbody>
 <?php
 $relations =
R::find('relationships','follower_id = ?',[$_GET['id']]);
 foreach($relations as $i){
 $user = R::findOne('users','id =
?',[$i['followed_id']]);
 ?>
 <tr>
 <td><?php echo $i['followed_id'] ?></td>
 <td><a
href="profile.php?id=<?=$i['followed_id']?>" class="btn btnsuccess"><?=$user['name']?></a></td>

 </tr>
 <?php } ?>
 </tbody>
 </table>

```
